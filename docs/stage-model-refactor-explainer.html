Stage-specific model & thinking resolution

Observer, Reflector, and Dropper are now resolved independently — each can pick its own model and thinking level, with a fixed inheritance chain so nothing changes when you don't configure it.

01The change in one paragraph

Previously all three background memory stages resolved a single shared model and a single shared thinking level. The evals showed these are different jobs that want different models — Observer captures literal events cheaply, Reflector/Dropper make judgment calls. The refactor lets each stage resolve its own model and thinking level from config, while keeping the shared model as the fallback so existing configs behave identically.

The contract: with no stage config, every stage resolves to the shared model and model.thinking ?? "low" — byte-for-byte the previous behavior. The whole feature is opt-in and additive.

02Three jobs, three temperaments

The DeepSWE evals made it concrete: a single model/thinking setting was forcing a bad compromise. Each stage has a distinct role.

stage 1

Observer literal capture

An event-capture job, not a reasoning job. Records what happened in the session. Eval finding: works best cheap, literal, with thinking off or low — high thinking drifted toward abstract summaries and lost concrete test/failure evidence.

↑ semantic F1 when grounded
stage 2

Reflector judgment

Promotes observations into durable reflections and assigns coverage. This is a synthesis/dedup/compression call. May want slower judgment and a stronger model — but "high thinking helps the reflector" is still unproven and needs a reflector-only replay.

model-sensitive (GLM-5.2 deepest)
stage 3

Dropper compression

Prunes the active observation pool toward a token target once the reflector has run. A judgment/compression task — closer to reflection than literal observation, so it inherits the Reflector's settings by default.

idle until pool over target

03Before vs after

Same pipeline shape — the only thing that changed is where the model and thinking come from before each stage runs.

Before — one resolution for all

// consolidation-trigger.ts
const resolved = await runtime.resolveModel(ctx);  // shared, cached once

// every stage reused the same values:
runObserver({ thinkingLevel: config.model?.thinking ?? "low" })
runReflector({ thinkingLevel: config.model?.thinking ?? "low" })
runDropper({   thinkingLevel: config.model?.thinking ?? "low" })

One model, one thinking — no way to give the observer cheap capture and the reflector deeper reasoning.

After — resolved per stage

// makeModelResolver: caches per stage
const desired = resolveStageModel(config, stage);
const resolved = await runtime.resolveModel(ctx, desired.model);

// each stage gets its own thinking:
runObserver({  thinkingLevel: resolved.thinking })  // off
runReflector({ thinkingLevel: resolved.thinking })  // high
runDropper({   thinkingLevel: resolved.thinking })  // inherits high

Each stage resolved independently; dropper inherits reflector unless overridden.

04The config schema

Three optional fields join the existing shared model. Each stage accepts an optional model override and an optional thinking level. Everything is optional.

export type StageName = "observer" | "reflector" | "dropper";

export interface StageModelConfig {
  model?: ConfiguredModel;     // { provider, id, thinking? } for this stage only
  thinking?: ModelThinkingLevel;  // off | minimal | low | medium | high | xhigh
}

export interface Config {
  // ...existing thresholds...
  model?: ConfiguredModel;          // shared fallback for every stage
  observer?:  StageModelConfig;       // NEW
  reflector?: StageModelConfig;       // NEW  (dropper inherits this)
  dropper?:   StageModelConfig;       // NEW
  passive: boolean;
  debugLog: boolean;
}
Invalid config is ignored, not fatal. A stage that isn't an object, a model with an empty id, or an unknown thinking value is dropped during normalization — same rule as the existing top-level config.

05The inheritance chains

This is the heart of the design. Each unset value walks a fixed fallback chain. Read top-to-bottom: the first value that exists wins.

Why dropper inherits reflector, not observer. Dropping is a judgment/compression task, closer to reflection than literal observation. So when dropper is unset it follows the reflector — but it stays its own stage and can override both model and thinking independently.

06How resolution flows at runtime

On each consolidation run, the trigger builds one resolver function that resolves + caches per stage, then each stage calls it for itself.

Two things to notice: (1) the resolver now caches per stage instead of once, so observer/reflector/dropper each resolve once and reuse on retry; (2) runtime.resolveModel gained an optional configured param and falls back to this.config.model when omitted — that's the entire backward-compat hinge.

07The three pure resolvers

All inheritance logic lives in three exported functions in config.ts — pure, no Pi runtime, no registry. That's deliberate: the rules are unit-tested in isolation, independent of auth or model availability.

/** model to look up in the registry for a stage */
export function resolveStageModelConfig(config: Config, stage: StageName): ConfiguredModel | undefined {
  if (stage === "observer")  return config.observer?.model  ?? config.model;
  if (stage === "reflector") return config.reflector?.model ?? config.model;
  return config.dropper?.model ?? config.reflector?.model ?? config.model;
}

/** reasoning level for a stage */
export function resolveStageThinking(config: Config, stage: StageName): ModelThinkingLevel {
  const shared           = config.model?.thinking ?? "low";
  const reflectorThinking = config.reflector?.thinking ?? config.reflector?.model?.thinking ?? shared;
  if (stage === "observer")  return config.observer?.thinking  ?? config.observer?.model?.thinking  ?? shared;
  if (stage === "reflector") return config.reflector?.thinking ?? config.reflector?.model?.thinking ?? shared;
  return config.dropper?.thinking ?? config.dropper?.model?.thinking ?? reflectorThinking;
}

/** both as a pair */
export function resolveStageModel(config: Config, stage: StageName) {
  return { model: resolveStageModelConfig(config, stage), thinking: resolveStageThinking(config, stage) };
}

Note the dropper line in resolveStageThinking: an explicitly configured dropper.model.thinking wins before the reflector fallback — that ordering was the one subtlety worth getting right.

08Backward compatibility, proven

The guarantee is testable and tested. With no stage config, every stage resolves identically to the old shared path.

// tests/config.test.ts
it("falls back to the shared model and shared thinking when no stage config is set", () => {
  const config = baseConfig({ model: sharedModel("medium") });
  for (const stage of ["observer", "reflector", "dropper"] as const) {
    expect(resolveStageModel(config, stage)).toEqual({ model: sharedModel("medium"), thinking: "medium" });
  }
});
